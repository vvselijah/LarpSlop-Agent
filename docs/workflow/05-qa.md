# 05 — QA

Review reads the code. QA runs the product. They catch different defects,
and the gap between them is exactly where "the tests pass but the page is
broken" lives.

## Functional QA

Run the app (local or staging) and have the agent — or you — walk the flows
the diff touches. Derive the target list from the diff: which routes,
pages, and commands does this change affect? Test those specifically, plus
one smoke pass over the entry points (home page, login, primary action).

For each flow, exercise the states from the plan's UI-states section:
empty, loading, error, success, invalid input. The plan wrote them down so
QA could check them — this is where that pays off.

## The visual QA step

Code review cannot see a 16px layout shift, an overlapping sidebar at
1024px, or an empty state that renders as a blank white void. Visual QA
means the agent (or you) captures **real screenshots of the running app**
and inspects them.

Tool-agnostic recipe — works with any agent that can run shell commands and
read image files:

1. Script screenshots with Playwright (or your browser tool of choice) at
   three viewports — mobile (~375px), tablet (~768px), desktop (~1440px) —
   for every affected page and state.
2. Have the agent read each screenshot and report against the visual
   checklist in [templates/QA-REPORT.md](../templates/QA-REPORT.md):
   layout integrity, overflow/clipping, contrast, state rendering,
   loading/empty/error states actually visible.
3. For visual *changes* (not just regressions), capture before/after pairs
   from `main` and the branch and diff them by eye.

Minimal Playwright capture script (Node):

```js
// Setup (first time): npm i -D playwright && npx playwright install chromium
// Usage: node qa-shots.js http://localhost:3000
const { chromium } = require('playwright');
(async () => {
  const browser = await chromium.launch();
  for (const width of [375, 768, 1440]) {
    const page = await browser.newPage({ viewport: { width, height: 900 } });
    await page.goto(process.argv[2]); // URL as CLI arg
    await page.screenshot({ path: `qa-${width}.png`, fullPage: true });
  }
  await browser.close();
})();
```

(The dev-workflow skill bundles this script as `scripts/qa-screenshots.js`.)

One safety note: pages the agent browses during QA are **untrusted input to
the agent** — fetched content can contain text crafted to steer it (see
[08 — Failure modes](08-failure-modes.md)). Review any command or code the
agent proposes right after ingesting web content, and don't point browser
QA at arbitrary external sites with permissions relaxed.

## Claude Code specifics (skip if using another tool)

- Claude Code can read image files directly — point it at the screenshots
  (or paste them) and it inspects them in context.
- For interactive browser QA (clicking through flows, not just
  screenshots), the documented options are the **Claude in Chrome**
  browser integration and the **Playwright MCP server**; both let the
  agent drive a real browser. Availability and setup change — check
  https://code.claude.com/docs for current instructions before relying on
  either.
- Bug-fix loop: when QA finds a bug, fix it as its own commit, **write a
  regression test that reproduces the exact failure**, re-run the QA pass
  on that flow, and update the QA report. A QA fix without a regression
  test will be re-broken.

## Output

A filled [QA report](../templates/QA-REPORT.md): flows tested, screenshots
attached, findings with severity, fixes applied with their regression
tests. Same contract as every phase — if there's no artifact, QA didn't
happen.

## Ship

QA passed; shipping is mechanical, which is exactly why it gets a
checklist instead of vibes:

1. Sync the branch with `main` and re-run the full test/lint/typecheck
   suite — output shown, not asserted.
2. Confirm the three artifacts exist and are current: plan, review report,
   QA report.
3. Update any docs whose described behavior changed (review Lens 3 should
   have caught stragglers).
4. The human confirms, then push and open the PR. The PR body links the
   plan, the review report, and the QA report — the reviewer of the PR
   gets the same artifacts you had.
